# M14.5 Public Landing Page + Vercel Deploy

M14.5 ships a **public marketing landing page** that describes the project for
game teams, historical map builders, and SaaS consumers, plus a release command
that can **validate**, **publish to GitHub**, and **deploy to Vercel**.

It sits between the M14 license-audited beta packs and M15 era-aware geometry.

## What is in the page

Static site under:

```text
landing/
  index.html
  styles.css
  app.js
  vercel.json
```

The page covers:

- dual audiences (strategy-game seeds vs atlas / SaaS maps)
- scaffold → graph → politics → export pipeline
- honest quality tiers (`scaffold-baseline`, `curated-politics`, `period-geometry`)
- what we do **not** claim
- license policy (public-safe core, isolated ODbL, excluded GADM)
- beta status and get-started commands
- interactive hero choropleth snapshot (same demo GeoJSON; 1444 / 1836 / modern owners)
- link into the **interactive MapLibre demo** under `landing/demo/`

## Commands

```bash
# Validate only (safe default / CI)
uv run gpm release site --dry-run

# Validate + JSON summary
uv run gpm release site --dry-run --format json

# Commit landing changes, push current branch, production deploy
uv run gpm release site --push --deploy

# Full bootstrap on a fresh clone with no remote yet
uv run gpm release site \
  --ensure-repo \
  --repo-name global-province-map-template \
  --push \
  --deploy

# Preview deployment (not production alias)
uv run gpm release site --deploy --preview
```

### `gpm release site`

| Flag | Effect |
| --- | --- |
| `--landing-dir` | Override site root (default `<project>/landing/`) |
| `--dry-run` | Validate only; no git/gh/vercel side effects |
| `--ensure-repo` | If the git remote is missing, create a GitHub repo via `gh` and attach it |
| `--repo-name` / `--repo-owner` | Name and owner for `--ensure-repo` |
| `--private` | Create a private GitHub repo (default public) |
| `--remote` | Remote name (default `origin`) |
| `--push` | `git add landing/`, commit if needed, `git push` |
| `--branch` | Branch to push (default: current) |
| `--commit-message` | Override commit message |
| `--deploy` | `vercel deploy --yes` from `landing/` |
| `--preview` | Preview deploy instead of `--prod` |
| `--format text\|json` | Summary format |

Validation always runs. Missing required files or on-message content snippets
fail with exit code 1.

### Prerequisites

| Tool | Used for |
| --- | --- |
| `git` | `--push` / repo checks |
| `gh` | `--ensure-repo` (GitHub API via CLI) |
| `vercel` | `--deploy` (must be logged in or set `VERCEL_TOKEN`) |

```bash
gh auth status
vercel whoami
# or: export VERCEL_TOKEN=...
```

## Deploy model

- **Source of truth:** this repository’s `landing/` directory (not a separate
  marketing monorepo, unless you choose one via `--ensure-repo` on a fork).
- **Hosting:** Vercel static project with `landing/` as the deploy root when
  using `gpm release site --deploy` (the command runs `vercel` with
  `cwd=landing/`).
- **GitHub:** existing remotes are reused; `--ensure-repo` only creates when
  the named remote is absent. If the GitHub repo already exists, it is linked
  instead of recreated.

### Manual Vercel alternative

```bash
cd landing
vercel --yes --prod
```

Or link the GitHub repo in the Vercel dashboard with **Root Directory** =
`landing`.
